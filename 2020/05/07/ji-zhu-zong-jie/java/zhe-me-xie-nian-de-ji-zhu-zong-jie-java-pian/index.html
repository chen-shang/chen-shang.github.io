先要理解一门编程语言的特点,然后才去学习它的语法,最后再去熟悉它的工具,这里我们会对比多种编程语言展开叙述。

Java 语言的是一门面向对象的编程语言，既然有面向对象的编程语言，那肯定有面向其他的编程语言：面向过程的编程语言、面向函数的编程语言。其实计算机世界目前就分这三种类型，我们有一个专有名词叫编程范式来定义一门编程语言所属的阵营。

C语言 是面向过程阵营的老大，还有很多元老级别的编程语言例如 Frotran、Basic；Java语言 是面向对象阵营的翘楚，此外还有C++、C#、Object-c、swift；函数式编程中最古老的例子莫过于1958年被创造出来的Lisp了；

其实编程范式的根本区别在于编程思想不同，就跟你走社会主义和资本主义一样，都有一个思想来指导，比如马克思主义毛泽东思想。一门语言不是说他是面向过程的就写不出面向对象形式的代码了，你完全可以用c语言写出继承、封装、多态的代码，只不过相当麻烦罢了。另外，你以为你用Java写的就是面向对象形式的代码了吗？肯定不是，看看你方法里面充斥着的各种if else,各种while,各种for循环吧吧，你其实写的还是面向过程的代码，只不过就是把这些方法写到了一个类里面，和我用c语言把函数写到一个头文件中有什么区别呢？

所以，一些后现代的编程语言致力于将这几种编程范式杂糅到一起，例如 Scala. Scala的目的就是想整合面向对象和函数式编程于一身，包括后来的Java8+都是在这么玩儿，另外 Java8+ 也抄袭了不少 Scala和Guava的特性。

面向对象编程的特性

继承、封装、多态。背的已经不能再熟了。如果非要在说一个，那就是抽象，其实抽象是所有编程语言的共同特性。例如 汇编是对机器语言的抽象，C语言是对过程的抽象，Java是对对象的抽象。对于Java来说，抽象抽象抽出来之后就封装成了对象。

封装是抽象的手段,把相关的属性和方法封装到一个类里面，把处理问题的过程封装到一个方法里面，都是在进行抽象的过程，越抽象越难理解，因为越抽象越简单，简单到只有一两行代码，到时候你只能看到局部，而看不到整体，所以才越难理解。

封装的好处就是高内聚，相关功能都在一处，阅读和查找代码的时候一目了然，例如HashMap的实现里面,你肯定知道 put、get方法都在一处，不会跑到其他类里面去找。试想一下如果没有封装，你找代码估计只能靠全局搜索了

继承也是一种手段，是人们为了少些代码而发明的一种手段。试想一下，如果没有继承，针对每一类对象都要定义一个类，然后这些类里面相同的字段你要copy好几次，相同的方法你也要定义好几遍，何其麻烦！

继承的好处就是为了少写代码，保护键盘。同时也带来了一些其他意想不到的好，例如 继承是实现多态的手段之一。

多态就是一种事物的多种表现形态。多态的目的一样是为了少些代码，保护键盘。多态主要体现在调用对象方法的时候，是想如果没有多态，那么同样针对每一个对象都要写一个函数来处理，这是面向过程的语言解不了的痛。

多态是由继承+重写实现的，其实实现也很简单，所谓继承就是继承了父类的非私有属性和非私有方法，注意是非私有的，你可以想象成，子类对象创建的时候，先开辟出一段空间用于存放父类对象的属性和方法，然后在开辟一块空间，存放自己的属性和方法。如果子类和父类有相同的方法，就把自己的方法覆盖到父类里面，等找方法的时候，现在自己的空间中寻找，如果找不到再去父类的空间中寻找，找不到就报错，找到了就执行，一旦把子类赋值给父类对象的引用，则子类对象自己空间的东西就找不到了。其实根本原因在于 子类的方法表中会先复制一份父类的方法，然后子类的方法直接指向自己的实现，在调用函数的时候，肯定找具体的那个对象所在空间的方法区的方发表中的方法引用，然后在根据方法引用的指向，直接找到自己的方法区中的方法，然后压栈、出栈 执行该方法。

注意，子类是继承不了父类私有的属性和方法的。而且注意，我们没必要给子类定义和父类一样的属性。

Jvm是基于栈的还是基于寄存器的呢
这个要看Jvm的具体实现，一般Jvm是基于栈的例如 HotSpot ，此外还有jvm是基于寄存器的，例如Dalvik，也就是android的虚拟机就是基于寄存器的。所以不能说jvm就一定是基于栈的。
基于栈的你会看到class 文件中有很多入栈、出栈的操作，遇到操作数就压栈，遇到指令去出栈，cpu计算完成后继续压栈。一个指令往往比较短，但是篇幅比较长。
基于寄存器的你会看到class文件中有很多常常的指令，因为jvm总是来回倒腾那几个寄存器(其实也是内存)的值，一个指令往往入参比较多，所以篇幅就比较短。
但最总都要吧操作数送到cpu，而cpu铁定是基于寄存器实现的，而且寄存器就那么几个。

内存溢出 out of memory ，其实应该叫做内存不足，对于栈的内存溢出分两种，一种是普通的内存溢出，一种叫栈溢出。 普通的内存溢出指的是，内存空间没有足够的空间给你申请了，这时候要么发生GC要么直接告诉你 out of memory. 当然栈和堆都是内存，所有都会在申请空间的时候发生内存溢出，也就是 out of memory error. 但是对于 栈来说，如果压栈的太多导致的内存空间不足我们叫做 stack over flow 也就是栈溢出。所以说栈溢出是内存溢出的一种特例吧。

内存溢出肯定是因为没有空间分配了，空间分配不出来是因为，大多数空间都被占用着。如果有很多内存被占用着，但其实有一些占用着的空间其实可以被用来分配空间，那么我们就叫做发生了内存泄漏，内存泄漏如果发生，随着时间的推移，内存会被吃的越来越多，最后导致没有空间分配，而发生out of memory 或者 statck over flow. 也就是说内存泄漏会导致内存溢出。如果是堆里面的溢出就叫 out of memory 如果是栈里面的内存溢出，就要看具体是因为什么原因到值的栈空间不够，如果是栈空间被不停的压栈压栈而导致的我们起了一个特殊的名字叫栈溢出 statck over flow,否则就叫普通的out of memory.

StackOverflowError 都明确说了是Stack的error所以只会跟栈有关系，而jvm一般有两个栈，一个叫虚拟机栈用来执行你程序中的方法的，一个叫本地方法栈，是当你调用了一个native方法之后才会用到的栈，当然也可能有的jvm实现只有一个栈，不区分本地方法栈。over flow 叫溢出嘛，顾名思义就是我不停的往里放，直到再也放不了了，就好像一个杯子，我不停的在里面加水，一段时间后水流出来了，就叫溢出。

out of memory 叫内存不足，也很明确就是跟 memory有关的都会发生内存不足，那栈是内存么，当然是，所以栈也会发生 内存不足。都说了是不足了，就是我管你要，你没有了叫不足。也就是我申请空间，你告诉我空间不够了，你异常吧。如果空间不足了，说明其他东西占据了空间，那么大概率是内存泄漏了，内存泄漏就是有些内存占着茅坑不拉屎了，你还释放不掉他。或者你申请的空间太大了，超过了内存的限制，例如你申请一个1万个元素的数组

-Xss 指的是每一个线程分配的栈空间大小，如果设置的太小，就容易发生 stack over flow ,但是 线程就可以创建的越多，如果设置的太大，就栈能存的栈帧就越多，也就是越能调用更多的方法。但是能同时运行的线程数就越少，因为栈空间总是有限的，不会分配给jvm太多的。

数据类型

  1. 浮点数
    在计算机系统的发展过程中，曾经提出过多种方法表达实数。典型的比如相对于浮点数的定点数（Fixed Point Number）。在这种表达方式中，小数点固定的位于实数所有数字中间的某个位置。
    还有一种提议的表达方式为有理数表达方式，即用两个整数的比值来表达实数。 定点数表达法的缺点在于其形式过于僵硬，固定的小数点位置决定了固定位数的整数部分和小数部分，不利于同时表达特别大的数或者特别小的数最终，绝大多数现代的计算机系统采纳了所谓的浮点数表达方式。这种表达方式利用科学计数法来表达实数，即用一个尾数（Mantissa，尾数有时也称为有效数字——Significand；尾数实际上是有效数字的非正式说法）一个基数（Base），一个指数（Exponent）以及一个表示正负的符号来表达实数。比如 123.45 用十进制科学计数法可以表达为 1.2345 × 102
    ，其中 1.2345 为尾数，10 为基数，2 为指数。浮点数利用指数达到了浮动小数点的效果，从而可以灵活地表达更大范围的实数
    实际上，位数一致的情况下，表示的数字的个数肯定是一样的。比如32位的数，考虑有一位数字不同就是不同的数字，那么能表示的数字个数就是 个，不管浮点定点都是一样多的。但是表示的数字范围和精度上就有差异了。定点数的小数点靠左，显然精度就高（小数点后的数字多），但能表示的范围就小（整数部分少）。同理，小数点靠右，精度就低，但表示的数字范围大。对浮点数来说，平衡了精度和表示范围的问题。
    同样的数值可以有多种浮点数表达方式，比如上面例子中的 123.45 可以表达为 12.345 × 101，0.12345 × 103 或者 1.2345 × 102。因为这种多样性，有必要对其加以规范化以达到统一表达的目标。在 IEEE 标准中，浮点数是将特定长度的连续字节的所有二进制位分割为特定宽度的符号域，指数域和尾数域三个域，其中保存的值分别用于表示给定二进制浮点数中的符号，指数和尾数。这样，通过尾数和可以调节的指数（所以称为”浮点”）就可以表达给定的数值了。
